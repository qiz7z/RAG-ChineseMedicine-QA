# ETL 解析修复技术文档

> 详细记录药典 Word 文档解析器的修复过程，包括段落样式覆盖扩展和 OCR 字符纠正。

---

## 一、问题背景

### 1.1 数据源

- **文档**：`data/raw/2020年药典一部.docx`
- **文档规模**：54,442 个段落
- **预期药品条目**：2,000+ 个

### 1.2 初始问题

初始解析器仅识别 `Heading #3` 和 `Heading #4` 两种样式作为药品名边界，导致：

- 药品条目仅 ~800 个（远低于预期）
- 15 种核心药材（黄芪、甘草、茯苓、麻黄等）完全缺失
- 检索 Hit@5 仅 46%

### 1.3 根因分析

通过检查原始 Word 文档，发现药品名使用的段落样式非常多样：

| 段落样式 | 使用场景 | 之前是否识别 |
|---------|---------|------------|
| `Heading #3\|1` | 大部分成方制剂 | ✅ |
| `Heading #4\|1` | 子剂型、部分主条目 | ✅ |
| `Body text\|5` | 人参等药材 | ✅（阶段二新增） |
| `Heading #1\|1` | 一捻金等 | ✅（阶段二新增） |
| `Normal` | 部分药品名 | ✅（阶段二新增） |
| **`Body text\|4`** | 甘草、茯苓、麻黄、柴胡、地黄、泽泻、附子、黄芪、黄芩 | ❌ |
| **`Body text\|6`** | 川木通 | ❌ |
| **`Heading #2\|1`** | 山药、山豆根、山茱萸 | ❌ |
| **`Heading #5\|1`** | 半夏、半边莲 | ❌ |

## 二、样式覆盖扩展

### 2.1 修改文件

`src/etl/parser.py`

### 2.2 修改内容

**修改前**：

```python
EXTRA_DRUG_NAME_STYLES = {'Body text|5', 'Heading #1|1', 'Normal'}
```

**修改后**：

```python
EXTRA_DRUG_NAME_STYLES = {
    'Body text|4', 'Body text|5', 'Body text|6',
    'Heading #1|1', 'Heading #2', 'Heading #5',
    'Normal',
}
```

### 2.3 验证方法

对每个新增样式，检查药品名段落的后随上下文，确认后跟拼音名（纯 ASCII 文本），验证为药品条目而非正文片段。

**示例验证**（以"甘草"为例）：

```
[3067] Body text|1  | 【贮藏】 置阴凉干燥处，防潮，防蛀。
[3068] Body text|4  | 甘 草                              <<< 药品名
[3069] Body text|3  | Gancao                             <<< 拼音名
[3070] Heading #4|1 | GLYCYRRHIZAE RADIX ET RHIZOMA      <<< 拉丁名
[3071] Body text|1  | 本品为豆科植物甘草Glycyrrhiza...    <<< 来源正文
```

### 2.4 启发式校验逻辑

新增样式不会直接被识别为药品名，而是通过 `_is_entry_boundary` 方法中的以下条件链进行校验：

```python
# 额外样式：Body text|4, Body text|5, Body text|6, Heading #2, Heading #5 等
if block_idx >= 0:
    for style_pattern in EXTRA_DRUG_NAME_STYLES:
        if style_pattern in style_name:
            if self._is_likely_drug_name(text) and self._next_is_pinyin(block_idx):
                return 'main'
            break
```

**两个条件必须同时满足**：
1. `_is_likely_drug_name(text)`：文本像药品名（含中文、2~15字、非章节标记等）
2. `_next_is_pinyin(block_idx)`：后续段落为拼音名（纯 ASCII）

这种双重校验有效避免了将正文片段误判为药品名。

## 三、OCR 字符纠正

### 3.1 问题发现

在验证修复后的数据时，发现部分核心药材仍未出现：

- **黄芪**：搜索"黄芪"无结果，但搜索"黄茂"有结果
- **黄芩**：搜索"黄芩"无结果，但搜索"黄苓"/"黄苔"有结果
- **川芎**：搜索"川芎"无结果，但搜索"川弯"/"川夸"/"川菖"有结果

进一步检查发现：**芪、芩、芎 三字在整个 54,442 段文档中完全不存在**，说明是 OCR 识别阶段的系统性错误。

### 3.2 OCR 错误映射表

**修改文件**：`src/etl/cleaner.py`

新增 `DRUG_NAME_OCR_FIXES` 映射表：

```python
DRUG_NAME_OCR_FIXES = {
    '黄茂': '黄芪',       # 芪→茂（字形相近）
    '黄苓': '黄芩',       # 芩→苓
    '黄苔': '黄芩',       # 芩→苔
    '川弯': '川芎',       # 芎→弯
    '川夸': '川芎',       # 芎→夸
    '川菖': '川芎',       # 芎→菖
    '川萼': '川芎',       # 芎→萼
    '主茯苓': '土茯苓',   # 土→主
}
```

### 3.3 字形相似性分析

| 正确字 | Unicode | OCR 错误字 | Unicode | 相似度说明 |
|-------|---------|-----------|---------|-----------|
| 芪 | U+82AA | 茂 | U+8302 | 上部"艹"相同，下部"氏"→"戊" |
| 芩 | U+82A9 | 苓 | U+82D3 | 上部"艹"相同，下部"今"→"令" |
| 芩 | U+82A9 | 苔 | U+82D4 | 上部"艹"相同，下部"今"→"台" |
| 芎 | U+828E | 弯 | U+5F2F | 整体字形相近 |
| 芎 | U+828E | 夸 | U+5938 | 部分笔画相似 |
| 芎 | U+828E | 菖 | U+83D6 | 上部"艹"相同 |
| 芎 | U+828E | 萼 | U+843C | 上部"艹"相同 |
| 土 | U+571F | 主 | U+4E3B | 仅一点之差 |

### 3.4 纠正应用位置

在 `clean_entry` 函数中，对 `drug_name` 和 `parent_drug` 均应用 OCR 纠正：

```python
# 清洗药品名
drug_name = entry.get('drug_name', '')
drug_name = drug_name.replace(' ', '').replace('\u3000', '')

# 应用 OCR 药品名纠正（黄茂→黄芪等）
for ocr_wrong, correct in DRUG_NAME_OCR_FIXES.items():
    if ocr_wrong in drug_name:
        drug_name = drug_name.replace(ocr_wrong, correct)

entry['drug_name'] = clean_text(drug_name)
```

## 五、注意事项

1. **OCR 纠正仅应用于药品名**：`DRUG_NAME_OCR_FIXES` 仅在 `clean_entry` 中对 `drug_name` 和 `parent_drug` 字段应用，不影响正文内容中的相同字符。这是因为在正文中，"黄苓"可能出现在复方制剂的处方组成中，如果全局替换可能导致上下文不一致。

2. **样式匹配使用 `in` 运算符**：`_is_entry_boundary` 方法中使用 `style_pattern in style_name` 而非精确匹配，因为 Word 文档的样式名可能包含后缀（如 `Body text|4` vs `Body text|4|1`）。

3. **双重校验避免误判**：新增样式识别依赖 `_is_likely_drug_name` + `_next_is_pinyin` 双重校验，确保只有真正的药品名条目被识别，而非使用相同样式的正文段落。
