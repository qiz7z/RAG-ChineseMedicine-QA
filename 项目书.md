## 1. 项目背景与意义

### 1.1 行业背景

《中国药典》是国家药品标准的法典，收载品种的药品标准为国家药品标准。药典内容涵盖中药材、中药饮片、提取物、成方制剂及单味制剂等，是药品生产、经营、使用、检验和监督管理部门共同遵循的法定依据。

2020年版《中国药典》一部收载中药材、中药饮片、植物油脂和提取物、成方制剂和单味制剂等，共计约 **2,700 余个品种**，内容涉及性状描述、显微鉴别、理化鉴别、含量测定、功能主治、用法用量等专业领域。

### 1.2 现有痛点

| 痛点 | 描述 |
|------|------|
| **检索效率低** | 传统方式通过纸质书或PDF查找药品信息，需逐页翻阅，耗时耗力 |
| **信息分散** | 同一药品的不同属性（性状、鉴别、含量测定等）分布在多个章节，难以快速获取全貌 |
| **专业门槛高** | 药典使用大量专业术语和化学名称，非专业人员难以理解 |
| **跨品种查询困难** | 如"哪些药材含有黄芪甲苷"、"含挥发油的药材有哪些"等横向查询几乎无法通过传统检索实现 |
| **更新追踪成本高** | 药典定期更新，版本对比和变化追踪需要大量人工 |

### 1.3 项目意义

构建基于 RAG（Retrieval-Augmented Generation，检索增强生成）技术的药典智能问答系统，具有以下意义：

1. **提升检索效率**：将"人找信息"转变为"信息找人"，用户通过自然语言即可获取药典信息，检索时间从分钟级降至秒级。
2. **保证回答准确性**：RAG 架构以药典原文为知识来源，大模型仅做理解和归纳，最大限度降低幻觉风险，确保回答可溯源。
3. **支持复杂查询**：支持跨品种、跨章节的语义检索和综合问答，释放药典数据的深层价值。
4. **降低使用门槛**：自然语言交互降低药典使用的专业门槛，使药师、研究人员、医药学生均能高效利用药典。
5. **为后续应用奠基**：系统可作为药品质检、用药指导、药典修订辅助等上层应用的底层知识引擎。

### 2.5 数据特点总结

1. **结构化程度较高**：层级清晰（药典→分类→药品→章节），适合结构化解析。
2. **数据量大**：约405万字符，属于中等规模，向量库选型可偏轻量级。
3. **专业术语密集**：包含大量化学名词（如"黄芪甲苷"、"阿魏酸"）、药理术语、分析化学方法，对 Embedding 模型的中文专业语义理解能力要求高。
4. **表格信息重要**：986个表格主要为色谱洗脱条件和含量测定数据，需要单独处理并关联到所属药品。
5. **存在OCR噪声**：需在数据清洗阶段进行系统性的文本纠错。

## 3. 项目目标

### 3.1 总体目标

构建一个基于《中国药典》2020年版一部的智能问答系统，用户可通过自然语言提问，系统从药典中精准检索相关内容，由大语言模型生成准确、可溯源的回答。

### 3.2 功能目标

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F-01 | 自然语言问答 | 支持用户用自然语言提问，如"人参的性味归经是什么" | P0 |
| F-02 | 精准检索 | 支持按药品名称、章节（性状/鉴别/含量测定等）精确检索 | P0 |
| F-03 | 回答溯源 | 每条回答标注来源（药品名称、章节、原文片段） | P0 |
| F-04 | 多轮对话 | 支持上下文关联的多轮追问 | P1 |
| F-05 | 跨品种查询 | 支持"含有某某成分的药材有哪些"类横向查询 | P1 |
| F-06 | 全文检索 | 提供传统关键词搜索作为补充 | P1 |
| F-07 | 模糊匹配 | 支持药品别名、俗称匹配（如"黄芪"↔"黄耆"） | P2 |
| F-08 | 管理后台 | 数据更新、检索调优、问答日志管理 | P2 |

### 3.3 性能目标

| 指标 | 目标值 |
|------|--------|
| 单次问答响应时间 | ≤ 5 秒（含检索+生成） |
| 检索 Recall@5 | ≥ 90%（对标准测试集） |
| 回答准确率 | ≥ 95%（回答内容与药典原文一致） |
| 幻觉率 | ≤ 3%（生成药典中不存在的信息） |
| 并发支持 | ≥ 20 QPS |
| 系统可用性 | ≥ 99.5% |

### 3.4 非功能目标

- **安全性**：药典为公开标准数据，无数据隐私问题；但系统应防止 Prompt 注入等攻击。
- **可维护性**：支持药典更新后增量导入新数据。
- **可扩展性**：架构支持后续接入药典二部（化学药）、三部（生物制品）和四部（通则）。

### 4.3 核心设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 切片策略 | 以"药品+章节"为粒度的结构化切分 | 药典数据结构清晰，按章节切分可保证语义完整性，避免跨药品内容混淆 |
| 检索策略 | 混合检索（向量+BM25+元数据过滤） | 药典中大量精确术语（化学名、限度值）需关键词精确匹配，语义检索负责概念关联 |
| 排序策略 | RRF融合 + Cross-Encoder Rerank | 先粗排召回Top-30，再精排取Top-5送入LLM，平衡效率与精度 |
| 元数据设计 | 药品名、分类、章节名、页码 | 支持按药品名/章节精确过滤，提升检索精度 |
| 防幻觉机制 | 严格Prompt约束 + 回答后校验 | 医疗领域对准确性要求极高，需多重保障 |

## 5. 详细技术方案

### 5.1 数据处理层

#### 5.1.1 数据解析

从 `.docx` 文件中提取结构化内容，处理流程如下：

```python
# 解析流程伪代码
1. 使用 python-docx 读取文档
2. 遍历所有段落和表格，保持原始顺序
3. 识别药品名称（Heading #3 样式 或 特定模式匹配）
4. 识别章节标记（【性状】、【鉴别】等）
5. 将表格关联到其所属的药品和章节
6. 输出结构化 JSON
```

**输出数据结构设计**：

```json
{
  "drug_id": "yc_00001",
  "drug_name": "人参",
  "drug_name_aliases": ["棒槌", "园参"],
  "category": "药材和饮片",
  "page_number": 1,
  "sections": [
    {
      "section_name": "来源",
      "section_content": "本品为五加科植物人参Panax ginseng C. A. Mey.的干燥根和根茎...",
      "table": null
    },
    {
      "section_name": "性状",
      "section_content": "主根呈纺锤形或圆柱形，长3～15cm，直径1～2cm...",
      "table": null
    },
    {
      "section_name": "含量测定",
      "section_content": "照高效液相色谱法（通则0512）测定...",
      "table": {
        "headers": ["时间（分钟）", "流动相A（%）", "流动相B（%）"],
        "rows": [["0~25", "28", "72"], ["25~33", "28—45", "72—55"]]
      }
    }
  ]
}
```

#### 5.1.2 数据清洗

| 清洗项 | 方法 | 示例 |
|--------|------|------|
| OCR字符纠错 | 建立错别字映射表，正则批量替换 | 【検查】→【检查】，【含量測定】→【含量测定】 |
| 全角/半角统一 | 统一数字和标点为标准格式 | `０．５`→`0.5` |
| 多余空白 | 合并连续空格和换行 | 多个空格→单个空格 |
| 页眉页脚 | 基于位置和模式匹配移除 | "中国药典2020年版一部" 等重复文本 |
| 段落合并 | 同一章节内连续段落合并 | 将被错误分段的内容重新拼接 |

#### 5.1.3 智能文本切片

药典数据的切片策略是 RAG 效果的关键。采用 **三级切分策略**：

```
第一级：按药品条目切分（每个药品为一个 Document）
    │
    ├── 短药品（< 2000字符）：整条存为一个 Chunk，不切分
    │
    └── 长药品（≥ 2000字符）：进入第二级切分
            │
            第二级：按【章节】切分（【性状】为一个 Chunk，【鉴别】为另一个...）
                │
                ├── 单章节 < 1500字符：整段存为一个 Chunk
                │
                └── 单章节 ≥ 1500字符：进入第三级切分
                        │
                        第三级：按自然段落滑动窗口切分
                        （窗口500字符，重叠100字符）
```

**每个 Chunk 的元数据结构**：

```json
{
  "chunk_id": "yc_00001_xzxz_001",
  "content": "主根呈纺锤形或圆柱形，长3～15cm...",
  "metadata": {
    "drug_name": "人参",
    "drug_id": "yc_00001",
    "category": "药材和饮片",
    "section": "性状",
    "page_number": 1,
    "chunk_index": 0,
    "total_chunks_in_section": 1
  }
}
```

#### 5.1.4 预估数据规模

| 指标 | 估算值 |
|------|--------|
| 药品条目数 | ~2,700 |
| 切片后 Chunk 总数 | ~15,000 - 20,000 |
| 平均 Chunk 长度 | ~250 字符 |
| 向量维度 | 1024（BGE-large-zh） |
| 向量库总大小 | ~80 MB |
| 全文索引大小 | ~50 MB |

> 结论：数据规模属于轻量级，Chroma 足以满足需求，无需分布式向量库。

### 5.2 索引构建层

#### 5.2.1 向量索引

- **Embedding 模型**：`BAAI/bge-large-zh-v1.5`
  - 中文语义理解能力强，在 C-MTEB 基准上表现优异
  - 输出维度 1024，支持最大 512 token 输入
  - 开源免费，可本地部署
- **索引类型**：HNSW（Hierarchical Navigable Small World）
  - 适合中小规模数据集
  - 查询延迟 < 10ms
- **存储**：Chroma 持久化到本地磁盘

#### 5.2.2 关键词索引

- **引擎**：Elasticsearch 或 Rank BM25（轻量级方案）
- **分词器**：IK Analyzer（中文分词）+ 自定义药典词典
- **自定义词典内容**：
  - 药品名称（人参、黄芪、丹参...）
  - 化学成分名（黄芪甲苷、阿魏酸、人参皂苷...）
  - 分析方法术语（高效液相色谱法、薄层色谱法...）

#### 5.2.3 元数据索引

在 PostgreSQL/SQLite 中建立药品维表，支持：

```sql

SELECT chunk_id FROM chunks WHERE drug_name = '人参';

-- 通过分类+章节联合过滤
SELECT chunk_id FROM chunks

```

#### 5.3.1 混合检索流程

```
用户提问: "人参的性味归经和功能主治是什么？"

    ▼
┌─────────────────────────────────┐
```

#### 5.3.2 RRF 融合算法

```python
# Reciprocal Rank Fusion 融合公式
def rrf_fusion(rank_lists, k=60):
    """
    rank_lists: 多路检索结果的排名列表
    k: 平滑参数，通常取60
    """
    scores = {}
    for rank_list in rank_lists:
        for rank, doc_id in enumerate(rank_list, 1):
            scores[doc_id] = scores.get(doc_id, 0) + 1.0 / (k + rank)
    return sorted(scores, key=scores.get, reverse=True)
```

#### 5.3.3 检索增强策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 元数据预过滤 | 先按药品名/章节过滤，再向量检索 | "人参的【含量测定】"等精确查询 |
| 查询扩展 | 同义词扩展（性味→性味与归经） | 用户用词与药典不一致时 |
| 多查询生成 | LLM 将一个问题拆分为多个子问题分别检索 | 复杂多跳问题 |
| 动态 Top-K | 根据问题类型调整返回数量 | 简单问题K=3，复杂问题K=8 |
